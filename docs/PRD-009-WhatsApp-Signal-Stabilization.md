# PRD-009: WhatsApp & Signal Stabilization & Bug Fix Plan

## Overview
This document outlines the root causes and remediation plan for the severe bugs identified in the "Merge-Unified-Messaging" project's WhatsApp and Signal integrations. 

### Identified Issues
1. **WhatsApp and Signal contacts are not showing, names are missing.**
2. **The "merge" feature for contacts is not working.**
3. **Messages are duplicated, not sending, or not loading correctly.**
4. **Group chats in WhatsApp and Signal are completely missing/not displayed.**

---

## 1. Root Cause Analysis

### Issue 1: Contacts Not Showing / Missing Names
* **Root Cause 1 (Signal & WhatsApp):** `App.tsx` uses a helper function `isProperName(name)` to filter incoming Signal contacts. This function explicitly skips names that are pure phone numbers (`/^\+?[\d\s-]+$/.test(trimmed)`). Because unsaved contacts in Signal and WhatsApp often default to just their phone number as their name, these contacts are entirely discarded from the state. 
* **Root Cause 2 (WhatsApp):** The fallback logic in `whatsapp-server.js` sometimes provides the phone number as the name if `pushname` is missing, which causes contact merging or display issues down the line.

### Issue 2: Contact Merging is Broken
* **Root Cause:** In `App.tsx` (`handleWhatsAppChatsLoaded`, `handleSignalChatsLoaded`), the logic attempts to cross-platform merge contacts based purely on exact string matching of `normalizeName(u.name)`. 
  * If a user is named "John D" on Signal and "John Doe" on WhatsApp, the strict match fails, resulting in duplicate entries.
  * The loop appending WhatsApp/Signal users to `prev` users array doesn't handle deduplication safely if the incoming batch contains previously cached contacts, leading to duplicated "unmerged" identities.

### Issue 3: Messages Duplicated, Not Sending, or Not Loading
* **Root Cause - Duplication (Signal):** Signal messages are deduplicated in `App.tsx` using a fingerprint based on `Math.floor(m.timestamp.getTime() / 5000)`. Because optimistic UI updates (when the user sends a message) have local timestamps, and the server sync echo has the exact server timestamp, if they fall across a 5-second boundary (e.g., 4999ms vs 5001ms), the fingerprint deduplication fails, causing the message to appear twice.
* **Root Cause - Not Loading (Signal History):** `signal-server.js`'s `getMessages` handler explicitly does nothing for fetching historical messages (it only returns what is cached in memory). Thus, if the app requests history not locally cached, nothing loads.
* **Root Cause - WhatsApp IDs mismatch:** The message IDs are formed differently between `whatsapp-server.js` and `ChatArea.tsx`, meaning the deduplication logic using exact `id` matches can fail if the prefix/suffix doesn't align exactly.

### Issue 4: Group Chats Missing Completely
* **Root Cause (WhatsApp):** In `App.tsx` inside `handleWhatsAppChatsLoaded`, there is an explicit line that completely skips group chats: `if (chat.isGroup) continue;`.
* **Root Cause (Signal):** If a Signal group has no explicit title, it falls back to names or IDs that trigger the strict `isProperName` regex filters (e.g., UUIDs), thereby silently dropping them.

---

## 2. Actionable Bug-Fix Plan

The coding sub-agents should execute the following tasks:

### Task 1: Fix Contact Filtering and Display
* **Location:** `Merge-Unified-Messaging/App.tsx`
* **Action:** 
  1. Remove or heavily loosen the `isProperName` filter. It must allow pure phone numbers (e.g., `+1234567890`) since that is the standard identifier for unsaved contacts.
  2. If the name is a UUID or missing, explicitly fallback to "Unknown Number: [Phone]" instead of dropping the contact entirely.

### Task 2: Fix WhatsApp and Signal Group Chats
* **Location:** `Merge-Unified-Messaging/App.tsx`
* **Action:** 
  1. Remove `if (chat.isGroup) continue;` in `handleWhatsAppChatsLoaded`.
  2. Map group metadata correctly (assigning roles like "WhatsApp Group" / "Signal Group" and ensuring the ID mapping works for group messaging, e.g., `@g.us`).

### Task 3: Overhaul Contact Merging Logic
* **Location:** `Merge-Unified-Messaging/App.tsx`
* **Action:**
  1. Instead of strict `normalizeName` matching, introduce phone-number matching. If `userA.id` or phone fallback matches `userB`, merge them.
  2. Ensure `setUsers` safely deduplicates incoming cached data vs live data so that multiple `onChatsLoaded` triggers don't clone entries. 

### Task 4: Fix Message Deduplication (Signal & WhatsApp)
* **Location:** `Merge-Unified-Messaging/App.tsx` & `Merge-Unified-Messaging/src/components/ChatArea.tsx`
* **Action:** 
  1. Replace the unreliable 5-second boundary fingerprint in `handleSignalMessagesLoaded` with a sliding window check (e.g., `Math.abs(existing.timestamp - new.timestamp) < 5000`) or rely strictly on a unified message ID schema generated by the server.
  2. Ensure the optimistic message ID generated in `ChatArea.tsx` strictly mirrors the logic the backend uses to identify a sent message, so Zustand deduplication works flawlessly via exact `message.id`.

### Task 5: Implement Signal History Fetching Fallback
* **Location:** `Merge-Unified-Messaging/server/signal-server.js`
* **Action:**
  1. If `getMessages` is requested, and `signal-cli` does not support deep history, read from the persisted `data-signal/messages` sqlite/JSON cache file properly to return the local historical record instead of breaking.
